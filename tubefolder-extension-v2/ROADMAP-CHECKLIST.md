## 우선순위와 근거

4단계로 나눴습니다. 뒷 단계로 갈수록 "먼저 안 정해두면 나중에 갈아엎어야 하는 정도"는 낮아지고, 대신 복잡도·리스크는 올라갑니다.

| 단계 | 기준 |
|---|---|
| 1단계 | 구조에 영향 — v2를 짜기 시작하기 *전에* 정해야 나중에 재작업이 없음 |
| 2단계 | 핵심 신규 기능 — 사용자가 가장 체감하는 가치, 구조가 정해진 다음 바로 착수 가능 |
| 3단계 | 고급 기능 — 외부 서비스 연동 등 복잡도·리스크가 높아 핵심 기능이 안정된 뒤 진행 권장 |
| 4단계 | 마무리 다듬기 — 있으면 좋지만 없어도 동작에 지장 없는 항목, 여유 있을 때 처리 |

## 1단계 — 구조 설계 (착수 전 결정)

- [ ] **가상 스크롤 구조 설계**
  - 이유: 폴더 하나에 영상이 많을 때(실측: 8,000개에서 렌더링 중 약 113ms 멈춤) 성능 문제가 실제로 확인됨. 렌더링 방식은 v2 전체 UI 컴포넌트 구조의 기반이라, 나중에 붙이면 관련 코드를 거의 다시 짜야 함.
  - 복잡도: 중간

- [ ] **저장소 백엔드 결정 (localStorage/chrome.storage.local 유지 vs IndexedDB 전환)**
  - 이유: GAP-ANALYSIS 로드맵 항목. 아래 "이어보기" 기능이 영상별 재생 위치 데이터를 추가로 저장해야 하고, 동기화 기능도 저장 구조에 영향을 주므로 백엔드를 먼저 정해야 나머지 기능 설계가 흔들리지 않음.
  - 복잡도: 중간

## 2단계 — 핵심 신규 기능

- [ ] **우클릭 메뉴에서 폴더 추가·이름변경·삭제 직접 처리** — 이번에 새로 요청됨
  - 배경: 현재(v1)는 유튜브 페이지 우클릭 메뉴에 폴더 목록만 뜨고, 폴더를 새로 만들거나 이름을 바꾸거나 지우려면 별도의 "폴더 관리" 화면(새 탭)을 열어야 함. 산들이 실제로 우클릭 메뉴를 쓰다가 발견한 불편함.
  - 목표: 우클릭 메뉴 안에서 바로 새 폴더 만들기 / 이름변경 / 삭제까지 처리 → 페이지를 벗어날 필요 없음
  - 구현 메모: Chrome의 우클릭 메뉴(`contextMenus` API)는 글자를 직접 입력하는 칸을 지원하지 않아서, "새 폴더 만들기"를 누르면 유튜브 페이지 위에 작은 입력창(미니 팝업)을 띄우는 방식이 필요함. 이름변경·삭제도 같은 방식. 폴더/영상 추가·이름변경·삭제 로직 자체는 v1에도 이미 있어서(storage.js), 이번 작업은 주로 **새로운 진입점(우클릭 메뉴 UI)을 추가하는 것**이지 데이터 구조를 바꾸는 게 아님 → 1단계 결정과 무관하게 병행 가능.
  - 복잡도: 중간 (구조 변경 없음, UI 진입점 추가 위주)

- [ ] **유튜브 재생목록 일괄 가져오기**
  - 이유: 처음 쓸 때 폴더를 하나하나 채우는 수고를 줄여줌(GAP-ANALYSIS ★★). 구조 변경 없이 추가 가능한 기능이라 먼저 처리.
  - 방식: YouTube Data API(키 필요, 50개/호출) 또는 키 없이 oEmbed 반복 호출
  - 복잡도: 중간

- [ ] **이어보기(재생 위치 기억) 기능**
  - 내용: 시청 중단 시점부터 이어서 재생
  - 주의: v1은 영상 클릭 시 유튜브로 새 탭을 여는 방식(`window.open`)이라 재생 위치를 알 수 없습니다. 이어보기를 구현하려면 **영상을 앱 안에 내장 재생(유튜브 IFrame Player API)하는 방식으로 전환**해야 하는데, 이건 v1 대비 꽤 큰 구조 변경입니다. v2 설계 단계에서 이 방식 전환을 반영할지 먼저 결정해야 합니다. 2단계 안에서는 구조 변경이 걸려있어 가장 나중으로 배치.
  - 복잡도: 높음 (구조 변경 수반)

## 3단계 — 고급 기능 (핵심 안정화 후 권장)

- [ ] **모바일 자동 동기화 (구글 드라이브 또는 원드라이브)**
  - 이유: GAP-ANALYSIS에서 가장 높은 우선순위(★★★)로 평가됐던 항목이지만, OAuth 인증·충돌 해결 로직이 들어가 리스크가 커서 핵심 기능이 안정된 뒤 진행을 권장합니다.
  - 참고: 기존 GAP-ANALYSIS 문서는 **구글 드라이브 appDataFolder** 방식을 분석해뒀습니다(무료·서버 불필요·OAuth 비민감 스코프). **원드라이브**는 이번에 새로 언급되어 아직 기술 검토가 안 된 상태 — 착수 시 OneDrive Graph API 방식도 함께 조사 필요.
  - 복잡도: 높음
