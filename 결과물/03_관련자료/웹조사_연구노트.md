# 웹조사 연구노트 (관련자료)

설계·고도화의 근거가 된 외부 자료 총조사 결과. 각 결론 옆에 **출처**를 둔다.

---

## 1. 저장소 — chrome.storage 한도
| 영역 | 수치 | 비고 |
|---|---|---|
| `storage.local` `QUOTA_BYTES` | **10 MB** (Chrome 113↓ 5 MB) | `unlimitedStorage` 권한으로 **한도 해제** |
| `storage.sync` | 100 KB 총량 / **8 KB·항목** / 512 항목 | 쓰기 120/분, 1800/시간 → 라이브러리엔 **너무 작아 미사용** |
| `storage.session` | 10 MB (Chrome 111↓ 1 MB) | 세션 한정 |

- `unlimitedStorage`는 한도·축출(eviction)에서 면제되나, 디스크 전역 한도 초과 시 오류 가능.
- 실무상 50 MB↑부터 느려지고 압박 시 축출 → 초대형은 IndexedDB 권장.
- **적용:** `unlimitedStorage` 추가(G1). 영상은 참조(노드 ~200B)라 1만 개도 ~2MB로 local에 충분.
- 출처: [chrome.storage 레퍼런스](https://developer.chrome.com/docs/extensions/reference/api/storage)

## 2. IndexedDB (대용량 대안)
- 용량 ~1 GB 또는 디스크 60%. **비동기**(메인스레드 비차단). Chrome은 큰 값에 Snappy 압축.
- **결론:** 초대형/메타 캐시 확장 시 정공법. `storage.js`의 `load/save`만 교체하면 마이그레이션 가능하게 캡슐화해 둠.
- 출처: [IndexedDB storage improvements](https://developer.chrome.com/docs/chromium/indexeddb-storage-improvements)

## 3. MV3 서비스워커 수명
- **전역 변수는 워커 종료 시 소실** → 상태는 storage에 저장.
- **이벤트 리스너는 최상위에서 동기 등록**(비동기 등록 시 이벤트 유실).
- `storage.onChanged`로 설정 변경 즉시 반영.
- **적용:** `background.js`는 리스너 최상위 등록 + 매번 storage 읽기(전역 상태 무의존) — 준수.
- 출처: [service worker lifecycle](https://developer.chrome.com/docs/extensions/develop/concepts/service-workers/lifecycle)

## 4. 탐색기 정렬 규칙
- **폴더가 파일과 분리되어 먼저** 정렬됨.
- 기본 **자연(숫자) 정렬**: "File 2" < "File 10"(문자 비교 아님). 레지스트리 `NoStrCmpLogical`로 토글.
- 자세히 보기 = 다중 열(이름/유형/크기 등).
- **적용:** `compareNodes`가 폴더 우선 + `localeCompare(...,{numeric:true})` 자연정렬.
- 출처: [Explorer 자연정렬](https://www.makeuseof.com/windows-file-explorer-change-numerical-sorting/) · [Explorer 정렬 옵션](https://www.digitalcitizen.life/transform-windows-explorer-filtering-options/)

## 5. 메타데이터 — oEmbed / noembed (API 키 불필요)
- `https://www.youtube.com/oembed?url=...&format=json` → `title`, `author_name`, `thumbnail_url`.
- ⚠️ **youtube oembed는 브라우저 fetch에서 CORS 차단** → `https://noembed.com/embed?url=...`(ACAO:* 허용)로 폴백.
- **적용:** `TubeStore.fetchMeta`가 oembed(확장 host_permission으로 직통) → 실패 시 noembed(웹/모바일). (G2)
- 출처: [YouTube oEmbed/CORS](https://abdus.dev/posts/youtube-oembed/) · [noembed](https://noembed.com/) · [oEmbed 활용](https://queen.raae.codes/2022-01-21-yt-oembed/)

## 6. YouTube Data API v3 쿼터 (선택적 재생목록 연동 시)
- `playlistItems.list` = **1 유닛/콜**(50개/콜), `playlistItems.insert` = **50 유닛/콜**.
- 기본 **10,000 유닛/일**(태평양시 자정 리셋). → 읽기는 넉넉(일 50만 항목), **쓰기는 일 ~200건** 제약.
- **결론:** 실제 재생목록 *읽기 가져오기*는 저렴(키·OAuth 필요). *양방향 쓰기*는 쿼터 부담 → 기본은 자체 가상 계층이 정답.
- 출처: [쿼터 계산기](https://developers.google.com/youtube/v3/determine_quota_cost) · [playlistItems.list](https://developers.google.com/youtube/v3/docs/playlistItems/list)
