# 튜브폴더 (Tube Folder) 프로젝트

## 프로젝트 목표
유튜브 영상을 폴더로 정리하고 드래그앤드롭·정렬 기능을 제공하는 Chrome 확장 프로그램.

## 기술 스택
- 언어: TypeScript
- 프레임워크: React
- 빌드 도구: Vite + CRX
- 형태: Chrome 확장 프로그램 (Manifest V3)

## 핵심 기능
- 유튜브 영상 폴더 관리
- 드래그앤드롭으로 영상 이동
- 폴더별 정렬 기능
- 유튜브 페이지 내 자연스러운 UI 통합

## 알려진 이슈 (작업 이력)
- MutationObserver 범위가 너무 넓어 성능 저하 발생
- Shadow DOM 재귀 탐색 비용 과다
- 6초마다 강제 전체 리스캔 문제
- 수정 대상 파일: playlistAddButton.ts, playlistCardScan.ts, playlistDom.ts

## 완전 자율 코딩 및 진화 원칙

### Claude Code 사용 권장
- 권장 대상: 파일 생성·수정, 버그 수정, 테스트 실행, 리팩토링 등 실제 코드를 다루는 모든 작업
- 권장 방식: 작업 전 "Claude Code에서 진행하시면 더 효율적입니다"라고 먼저 안내한다.
- 예외: 사용자가 이미 Claude Code에서 작업 중이거나 코드 설명·설계 논의인 경우 생략.

### 작업 시작 전 필수 절차
- 프로젝트 폴더 구조를 먼저 파악한 뒤 작업을 시작한다.
- 수정 대상 파일을 특정하고, 영향 범위를 먼저 분석한다.
- 테스트·리팩토링 전 반드시 현재 버전을 백업한다.
- 주요 변경 사유와 내용을 코드 주석 또는 커밋 메시지에 기록한다.

### 완료 판정 기준 (루프 종료 조건)
- 코딩 결과를 점검해 🔴(치명적)·🟡(중요) 결함이 0개이면 "완성"으로 판정한다.
- 남아 있는 🟢(선택적 개선)는 강제 반영하지 않고, 목록으로 투명하게 보고만 한 뒤 멈춘다.
- 즉, 코드를 다시 점검했을 때 새로운 🔴·🟡 결함이 나오지 않는 상태가 목표다.
- (현실적으로 🟢 개선거리는 항상 존재할 수 있으므로, 이를 0으로 만드는 것은 완료 조건이 아니다.)

### 무한루프 방지 및 중단 보고 규칙
아래 조건에 해당하면 즉시 작업을 중단하고 사용자에게 보고한다.

**중단 조건**
- 반복 횟수 초과: 동일 문제 해결 시도가 10회를 넘어도 완료 판정에 도달하지 못한 경우
- 답보 상태: 수정해도 같은(또는 유사한) 오류가 다시 발생해 개선이 정체된 경우
- 순환 모순: A를 고치면 B가 깨지고, B를 고치면 A가 깨지는 식으로 맴도는 경우
- 컨텍스트 한계: 컨텍스트 창이 80% 이상 소진되어 정확한 판단이 어려운 경우

**중단 시 보고 내용**
- 현재까지 몇 회 반복했는지
- 어떤 문제가 반복되고 있는지 (반복 원인)
- Claude가 보는 막힌 이유와 가능한 선택지 (기준 완화 / 보류 / 방향 전환 / 새 세션 시작)
- 사용자의 결정을 받은 뒤에만 재개한다.

### 적용 범위 및 경계선
- 자율 완결은 "이미 승인된 설계·기능 범위 안에서의 구현·테스트·최적화"에 한정한다.
- 아래 항목은 반드시 사용자 승인 후 진행한다.
  - 설계·기능 구조의 변경
  - 의존성 패키지 추가·제거
  - 데이터 삭제·파일 권한 변경 등 되돌리기 어려운 작업

### Git 커밋 기준
- 🔴·🟡 결함이 0개인 상태에서만 커밋한다.
- 커밋 메시지는 "무엇을 왜 변경했는지"를 명시한다.
- 대규모 리팩토링 전에는 별도 브랜치를 생성한다.

### 토큰 효율 규칙
- 파일 전체를 불필요하게 반복 출력하지 않는다.
- 변경이 필요한 부분만 정확히 수정한다.
- 긴 파일은 필요한 섹션만 읽고 작업한다.

### 앱 완성 후 사용자 매뉴얼 제작 규칙
- 앱 코딩·검증이 모두 완료되어 "완성" 판정이 나면, 사용자 매뉴얼 제작을 권장한다.
  → "앱이 완성되었습니다. 처음 사용하는 분을 위한 사용자 매뉴얼(PDF)을 제작할까요?"
- 사용자가 승인한 경우에만 매뉴얼 제작을 시작한다.

## 개발 중 개선안 제안 규칙
- 각 개발 단계 진행 중 더 나은 방법·구조·기능이 발견되면 즉시 사용자에게 제안할 것
- 제안 시 현재 방식과 개선안을 비교해서 설명하고, 사용자가 채택 여부를 결정
- 설계·기능 변경은 사용자 승인 없이 임의로 변경하지 않음
- 제안은 간결하게, 쉬운 말로 설명할 것
